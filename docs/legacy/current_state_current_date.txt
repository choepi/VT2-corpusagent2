============================================================
1. Executive summary
============================================================

Short version
- This repo is no longer a toy. It contains a substantial agent runtime, a reproducible offline pipeline, a FastAPI backend, a static frontend, Dockerized Postgres and OpenSearch services, deployment scripts, and a reasonably large automated test suite.
- The repo is also not one cleanly unified system. It currently contains three overlapping layers:
  1. an offline preprocessing/evaluation pipeline,
  2. an older deterministic framework/tool-registry stack,
  3. a newer agent-runtime "magic box" stack.
- The most advanced and actually live-usable subsystem right now is the newer agent runtime backed by OpenSearch and Postgres.
- The older deterministic framework is code-rich and test-covered, but in this environment it is blocked by missing retrieval/NLP artifacts and therefore currently produces "unsupported/unavailable" answers instead of real analyses.
- The current environment is not truly hybrid retrieval, despite the README/TOML language. Live retrieval is effectively lexical OpenSearch search plus Postgres document fetch plus rerank plus analysis nodes. Dense retrieval is not operational end to end.

Most important verified facts
- Postgres is live, healthy, and populated with 624,095 rows in `article_corpus`.
- OpenSearch is live, healthy, and populated with 624,095 indexed documents in `article-corpus-opensearch`.
- Postgres has zero dense embeddings populated in `article_corpus`.
- The local lexical asset directory `data/indices/lexical/` is missing, which breaks `main.py` and `scripts/07_mcp_server.py`.
- The local dense file `data/indices/dense/dense_embeddings.npy` exists, but it is not usable as a complete asset:
  - `dense_doc_ids.joblib` is missing,
  - the matrix shape is `(624095, 768)`,
  - only the first 2,048 rows are non-zero,
  - 622,047 rows are all-zero.
- `pytest -q tests` passes: 71 passed, 1 skipped.
- `pytest -q` from repo root fails locally because pytest descends into `deploy/data/pgdata` and hits permission issues. This is a local workspace hygiene problem, not necessarily a code failure.

Bottom-line thesis assessment
- There is enough here for a strong master's project, but the next phase should not be "add more tools first".
- The highest-value next phase is to make one end-to-end operating mode coherent, reproducible, and evaluated:
  - repair retrieval integrity,
  - eliminate config/doc drift,
  - build serious evaluation data,
  - then run controlled experiments on the agent/runtime claims.

Dense asset reality
- `data/indices/dense/dense_embeddings.npy` exists and has apparent size about 1.8G.
- Actual matrix shape: `(624095, 768)`, dtype `float32`.
- The file behaves like an incomplete or sparse/interrupted streaming write:
  - non-zero rows in first 50,000 entries: 2,048
  - non-zero rows in all later chunks: 0
  - total zero rows: 622,047
- `data/indices/dense/dense_doc_ids.joblib` is missing.
- Conclusion: the dense file should not be treated as a valid completed index artifact.

Lexical asset reality
- `data/indices/lexical/` is absent.
- Required files such as `tfidf_vectorizer.joblib`, `tfidf_matrix.joblib`, and `tfidf_doc_ids.joblib` are absent.
- Any code path that requires local lexical assets fails immediately.

Effective retrieval mode right now
- `config/app_config.toml` declares `pgvector` backend and `hybrid` default mode.
- The local `.env` overrides runtime retrieval to `local` backend and `lexical` default mode, with backend services required and local fallback disabled.
- Because Postgres has zero embeddings and local dense assets are incomplete, dense retrieval is not a real option in practice.
- Therefore the live system should be described as:
  - OpenSearch lexical retrieval,
  - Postgres document fetch / working set persistence,
  - optional reranking / downstream tool analysis,
  - not true dense+lexical hybrid retrieval.

- Large real corpus is present and normalized.

  - retrieval queries = 3,

11. Recommended next steps for the master's project
============================================================

Priority 1: make one operating mode coherent
- Choose the thesis target explicitly:
  - Option A: lexical OpenSearch + Postgres + agent-runtime tools
  - Option B: true hybrid retrieval with dense + lexical + fusion + rerank
- Do not keep both stories half-alive.
- Right now the repo tells both stories simultaneously and neither is fully clean.

Priority 2: repair retrieval end to end before deeper agent research
- Rebuild lexical assets cleanly and verify they load.
- Rebuild dense assets cleanly and verify:
  - non-zero rows for all documents,
  - presence of `dense_doc_ids.joblib`,
  - reproducible summary output with timestamps and validation stats.
- If using pgvector:
  - ingest embeddings into Postgres,
  - build IVFFlat/HNSW indexes,
  - verify dense retrieval returns sane results.
- Only after this should you run retrieval ablations.

Priority 3: harden temporal data
- Parse/store `published_at` in a real temporal type.
- Add a data-quality report for blank/unparseable dates.
- Decide how undated documents are handled in trend analyses.

Priority 4: build serious evaluation sets
- Expand retrieval queries far beyond 3 examples.
- Expand faithfulness claims far beyond 4 examples.
- Expand benchmark questions into a real evaluation set with categories, difficulties, and expected evidence characteristics.
- Add manual annotation where necessary.

Priority 5: decide the fate of the older framework
- Either:
  - fully restore its artifact path and keep it as a deterministic baseline,
  - or officially demote/deprecate it and stop pretending it is the main path.
- Right now it is architecturally interesting but operationally secondary.

Priority 6: clean up heuristic boundaries
- Explicitly mark which capabilities are thesis-core and which are convenience heuristics.
- Candidate thesis-core components:
  - retrieval,
  - working-set creation,
  - evidence table construction,
  - grounded synthesis,
  - provenance.
- Candidate convenience/prototype components:
  - entity linking placeholder,
  - claim strength heuristic,
  - quote attribution heuristic,
  - weak exploratory topic/entity scripts.

Priority 7: improve runtime robustness
- Mark interrupted runs as failed/aborted on startup or heartbeat timeout.
- Pre-pull the sandbox Docker image instead of pulling at query time.
- Add stronger sandbox result/error capture.
- Exclude Docker volume directories from pytest discovery.

Priority 8: fix repo hygiene and drift
- Rename `scripts/00_1_downlaod.py`.
- Align `scripts/22_prepare_vm_stack.py`, `docs/deployment.md`, `README.md`, and `config/app_config.toml`.
- Remove or clearly mark stale snapshot docs.
- Add validation scripts that compare:
  - config vs effective runtime,
  - summary files vs live services,
  - expected asset files vs actual asset files.

Suggested next deep-research framing
- The strongest next research contribution is not "more NLP tools".
- The strongest next contribution is:
  "Can a provenance-first agent runtime over a large news corpus deliver grounded longitudinal analysis, and what retrieval/verification design is actually necessary to make that reliable?"
- To answer that well, the immediate next engineering milestone is a coherent evaluated evidence substrate.
